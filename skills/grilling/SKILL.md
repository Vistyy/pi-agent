---
name: grilling
description: Interview the user relentlessly about a plan or design. Use when the user wants to stress-test a plan before building, or uses any 'grill' trigger phrases.
---

Grill the design.
Walk the design tree relentlessly, resolving dependent decisions one at a time until the shared understanding is explicit.
Turn fuzzy intent into explicit accepted, rejected, and unresolved decisions.

Keep a tiny visible ledger of accepted, rejected, and unresolved points.
Do not print the whole ledger every turn.
Show only entries relevant to the current point, and occasionally checkpoint the ledger before a pivot.

For each turn:

1. Update the ledger from the user's last answer.
2. If code or docs can answer the current point, investigate instead of asking.
3. Make the strongest defensible recommendation.
4. Explain the reason briefly.
5. Ask whether the user accepts it.

Only ask an open narrowing question when no defensible recommendation exists.
If so, explain why first.

Ask exactly one question.

Never repackage a rejected mechanism under a new name.
Do not generate proposals just to keep momentum.
